2.6	语义路由和负载均衡
2.6.1	项目背景
大模型服务已经从单模型、单集群、单场景的简单调用，快速演进到多模型、多供应商、多租户、混合部署、长上下文、多模态和复杂治理并存的阶段。在该背景下，传统 API 网关或传统四层、七层负载均衡方案面临明显不足。

传统 Web 服务的请求成本相对稳定，调度重点通常是连接数、请求数和实例健康状态；而在大模型推理场景下，一个请求的资源占用和执行代价受到以下因素显著影响：
•	输入上下文长度和历史会话长度差异极大；
•	输出长度与推理复杂度波动明显；
•	请求是否涉及工具调用、多模态、结构化输出、代码生成或长链推理；
•	模型实例运行态受 GPU 利用率、显存占用、队列深度、Prefill / Decode 阶段切换和缓存命中状态影响较大；

在大模型推理架构中，语义路由与负载均衡已不再是传统 API Gateway 的简单扩展，而是位于用户语义空间与异构算力集群之间的智能控制平面。其核心职责不只是请求分发，而是在复杂治理边界下，完成能力域选择、实例调度、故障切换与资源优化的联合决策。

在大规模语言模型（LLM）推理架构中，语义路由与负载均衡已由传统网关组件演进为位于用户意图与算力集群之间的解耦式智能控制平面（Control Plane）。该中枢不仅负责请求的语义解析与负载画像，更集成了安全审计、语义缓存、上下文记忆及工具编排等全栈控制面能力，形成基于推理反馈的策略闭环。其核心挑战已从单一的流量分发转向多目标约束下的实时联合优化：系统需在复杂约束下，实现从宏观模型池选择到微观执行实例、物理计算单元（GPU），乃至原子级推理阶段（Prefill vs. Decoding）的精细化资源映射。
2.6.2	产品目标
构建一个推理场景的语义路由与负载均衡子模块，使系统能够基于请求语义、资源状态与治理约束，在毫秒级完成模型选择与实例调度，并通过持续反馈实现质量、时延、成本与稳定性的联合优化。构建一个面向大模型推理场景的统一语义路由和负载均衡系统，使系统能够基于请求语义、资源状态和治理约束，在毫秒级完成模型选择与实例调度，并通过持续反馈实现质量、时延、成本和稳定性的联合优化。

2.6.4	整体架构
语义路由和负载均衡的设计目标是在请求进入执行链路之前，完成语义理解、边界裁剪、路径选择和运行态调度的分层决策。对于大模型请求而言，模型能力匹配、租户与治理约束、成本与时延权衡、实例运行态优化并不属于同一类问题，如果在同一层中混合处理，系统将难以保持清晰的职责边界，也难以支撑解释、审计和后续演进。
模块	任务	输入	输出
Semantic Routing	请求应该去哪个能力域/模型池	任务类型、上下文长度、租户策略、成本等级、风险等级	模型池、主备路径
Load Balancing	请求应该落到哪个实例/GPU	实例健康、队列深度、KV Cache、GPU 利用率	具体实例、执行节点

基于这一判断，系统采用四层结构：请求理解与状态构建层、候选池过滤层、池级路由层、实例调度层。四层之间遵循由抽象到具体、由静态约束到动态优化、由能力域判断到执行落点判断的递进关系。前两层负责形成正确且合法的决策空间，后两层负责在该空间内完成策略选择与运行态落点选择。
系统设计遵循以下原则：
•	语义正确性优先于运行时优化，先确定请求需要什么能力，再讨论如何优化执行路径。
•	治理约束优先于性能目标，任何时延、成本或吞吐优化都不得突破权限、风险和数据边界。
•	池级路由与实例调度严格分层，避免能力域选择与实例运行态选择相互污染。
•	静态信息与动态信息分层处理，模型能力、租户边界、治理规则等稳定信息尽量前置，实例负载、健康状态、缓存命中等高频信息留在后层处理。
•	决策链路必须具备解释性，能够清晰说明请求为何被裁剪、为何选择某个模型池、为何落到某个实例。
2.6.4.1 请求理解与状态构建层
该层负责将异构的北向请求转化为统一的内部决策语义。对系统而言，原始接口字段并不是最终决策对象，真正参与路由判断的是请求所表达的任务类型、复杂度、上下文规模、模态要求、会话连续性、风险特征以及调用偏好。因此，该层的首要职责是完成标准化和语义抽象，将接口差异消解为可供后续层稳定消费的统一事实集合。
在此基础上，该层进一步构建 Routing State，作为后续决策的统一上下文。Routing State 不只是请求内容的映射结果，还应整合租户身份、治理边界、策略偏好、会话状态和必要的环境信息。其设计意义在于，为整个决策链提供单一可信的输入源，避免不同模块重复解析请求并产生不一致结论。
从职责边界看，该层只负责形成“可决策请求”，不负责决定具体流向。它解决的是请求认知问题，而不是调度选择问题。将这两类问题分离之后，语义识别能力可以独立演进，策略调整也不会反向破坏请求理解逻辑。
2.6.4.2 候选池过滤层
该层负责基于 Routing State 定义合法的候选空间。其核心任务不是排序，而是边界裁剪。系统需要结合模型池画像、能力约束、租户边界、权限规则、风险规则和显式调用控制，对所有可达模型池进行一次严格过滤，仅保留在当前请求条件下具备合法性和兼容性的候选集合。
这一层的设计逻辑在于，将“是否允许进入某个能力域”的判断前置完成。窗口不足、模态不支持、权限不满足、数据等级不匹配、覆盖控制未授权等问题，本质上都属于硬约束，不应被后续的时延或成本偏好重新平衡。只有当候选域先被裁剪清楚，后续的路由优化才具备稳定边界。
因此，候选池过滤层是整个系统的边界收口层。它确保后续任何优化都只能发生在合法范围内，防止池级路由和实例调度承担本不属于它们的治理职责。对于无合法候选池的请求，系统应在此处直接给出明确结果，而不是将不确定性下推到后续链路。
2.6.4.3池级路由层
该层负责在合法候选池集合内完成能力域级别的路径选择。经过前两层处理后，系统已经明确请求的语义特征以及合法边界，此时需要回答的问题不再是“能否服务”，而是“在可服务范围内，哪一类模型池最适合作为主路径”。这一层面向的是质量、时延、成本、稳定性等策略目标之间的权衡。
池级路由层的关键设计点在于，它只处理模型池这一抽象层级，而不直接介入具体实例的瞬时调度。模型池代表相对稳定的能力域和策略域，例如高质量池、低时延池、长上下文池或特定供应商池；实例则体现高频变化的运行态。将二者分离，可以保证池级决策围绕能力匹配和策略优先级展开，而不被局部瞬时负载扰动。
因此，该层应输出主路径和备选路径，并保证备选路径与主路径处于同样合法的治理边界之内。治理优先级必须高于调用方偏好，主备路径必须服务于同一决策逻辑。这样，后续的异常切换才不会演化为越界调度。
2.6.4.4 实例调度层
第四层负责在目标模型池内部完成实例级执行落点选择。进入该层后，能力域已经确定，系统关注点从“选哪一类池”转向“在当前运行态下由哪个实例承接请求更合适”。这一层主要处理实例健康、并发压力、队列深度、资源利用率，以及会话亲和、缓存亲和、长短请求隔离等大模型场景下的运行时特征。
实例调度层的设计逻辑是，在不突破上层边界的前提下最大化执行效率与系统韧性。它应优先在既定模型池内部完成选择与重选，尽可能兼顾健康性、负载均衡和复用收益。当实例异常、过载或不可达时，应先在池内进行受控重选；只有在满足明确条件时，才可根据上层给定的备选路径执行进一步切换。
这一分层保证了两个层面的稳定性：一方面，上层已经确定的能力与治理判断不会被运行态策略改写；另一方面，运行态层可以独立围绕局部负载和故障情况做快速调整，而不影响池级路由的整体逻辑。
2.6.5	层间协同于控制
四层设计形成的是一条逐层收敛的决策链，而不是若干并列模块的简单拼接。第一层完成请求语义抽象，第二层定义合法候选空间，第三层在合法空间内选择主备路径，第四层在既定路径内确定具体执行落点。每一层都以消化本层问题为目标，不将未完成的职责向下游转移。
这种协同方式本质上是对决策问题进行两次解耦。第一次解耦是将语义与治理边界同运行态优化分开，保证请求先被正确理解、正确裁剪，再讨论性能最优。第二次解耦是将能力域选择同实例选择分开，保证池级决策关注稳定策略，实例决策关注实时状态。通过这两次解耦，系统可以在保证语义正确与治理可控的前提下，逐步引入更复杂的优化能力，而不会破坏整体结构。
异常路径同样遵循分层闭环逻辑。若请求在候选池过滤层已无合法去向，系统应直接终止决策并返回明确结果；若池级路由已产生主备路径，实例调度层只能在既定路径内进行受控重选与切换；若执行阶段发生异常，观测、审计和反馈机制应能够逆向定位问题发生于语义识别、边界裁剪、池级选择还是实例调度。由此形成从请求进入、决策执行到异常复盘的完整控制闭环。

2.6.8	总结
语义路由与负载均衡系统不是传统 API 网关或普通负载均衡器的简单增强版本，而是 LLM Gateway 面向大模型场景的核心控制面能力。它要解决的不只是“请求分发到哪里”，而是“什么请求在什么约束下应该进入什么模型池，并在当前运行态下落到哪个实例执行”。该系统将成为 Gateway 后续多模型接入、多租户治理、异构后端纳管、策略优化和运行效率提升的统一底座。它既决定系统的短期可用性，也决定平台的长期扩展能力和运营能力。
