# LLM Gateway 语义路由与负载均衡子系统需求规格说明书（SRS）

## 1. 文档说明

### 1.1 编写目的

本文档用于定义 `LLM Gateway` 内部“语义路由与负载均衡子系统”的建设目标、职责边界、核心能力、功能需求、非功能需求、接口契约、验收标准和迭代范围，作为产品、架构、研发、测试、运维和安全团队的统一评审依据。

本文档关注“系统应当建设什么、达到什么标准、以什么边界落地”，而不是具体实现细节。设计、开发、测试和验收阶段均应以本文档为上位约束，保证需求、方案和交付结果的一致性。

### 1.2 适用范围

本需求文档适用于以下范围：

- 作为 `LLM Gateway` 内部控制面的子系统，负责对大模型请求进行语义理解、候选池筛选、模型池选择、实例级调度、治理约束执行和观测反馈。
- 面向北向调用方提供统一的推理入口，对南向异构模型服务、供应商接口和实例集群提供一致的调度抽象。
- 一期建设以 `OpenAI API` 兼容请求为主，服务 `chat/completions` 等主流推理调用路径。

### 1.3 目标读者

本文档面向以下角色：

- 产品经理：用于确认业务目标、建设边界和优先级。
- 架构师：用于评估职责拆分、接口边界和系统演进路径。
- 研发工程师：用于实现路由、调度、治理和可观测能力。
- 测试工程师：用于制定功能、性能和稳定性验证方案。
- 运维和 `SRE`：用于建设运行保障、容量规划和故障处置机制。
- 安全与合规团队：用于评审权限控制、风险治理和审计要求。

### 1.4 术语与约定

本文档中使用下列关键词表示要求强度：

- `MUST`：必须满足，否则视为不符合需求。
- `SHOULD`：推荐满足，除非有明确理由暂缓。
- `MAY`：可选能力，可根据迭代优先级决定是否实现。

核心术语定义如下：

| 术语 | 定义 |
| --- | --- |
| 语义路由 `Semantic Routing` | 基于请求内容、调用上下文和治理约束，决定请求“允许进入哪些模型池、应优先进入哪个模型池”的控制面能力 |
| 负载均衡 `Load Balancing` | 在已选模型池内部，根据实例健康、负载、亲和和运行态信息选择具体执行实例的能力 |
| 模型池 `Target Pool` | 具有相近能力边界、成本档位、治理属性或供应商特征的一组模型或实例集合，是语义路由的主要调度对象 |
| 路由状态 `Routing State` | 将请求事实、控制参数、租户身份、治理约束、会话上下文和策略偏好归一化后的统一决策上下文 |
| `RouteDecision` | 语义路由阶段输出的模型池级决策结果 |
| `DispatchDecision` | 负载均衡阶段输出的实例级调度结果 |
| 显式路由控制 | 调用方通过 `model`、`routing_hint`、`route_override` 等字段表达的显式意图 |
| 会话亲和 | 在满足合法性和健康性约束前提下，使同一会话优先落在稳定实例或稳定实例集合上的调度策略 |
| `Prefix / KV Cache` 亲和 | 优先将具有相似上下文前缀的请求调度到具备更高缓存复用概率的实例上 |

## 2. 项目背景与问题定义

### 2.1 背景

大模型服务已经从单模型、单集群、单场景的简单调用，快速演进到多模型、多供应商、多租户、混合部署、长上下文、多模态和复杂治理并存的阶段。在该背景下，传统 API 网关或传统四层、七层负载均衡方案面临明显不足。

传统 Web 服务的请求成本相对稳定，调度重点通常是连接数、请求数和实例健康状态；而在大模型推理场景下，一个请求的资源占用和执行代价受到以下因素显著影响：

- 输入上下文长度和历史会话长度差异极大；
- 输出长度与推理复杂度波动明显；
- 请求是否涉及工具调用、多模态、结构化输出、代码生成或长链推理；
- 模型实例运行态受 GPU 利用率、显存占用、队列深度、`Prefill / Decode` 阶段切换和缓存命中状态影响较大；
- 企业级场景下还要同时满足租户隔离、权限控制、风险治理、审计追踪和成本控制要求。

因此，系统需要的不再是单纯“把请求均匀分发出去”，而是一套面向 `LLM Gateway` 的智能控制面，在执行前完成语义识别、合法候选筛选、模型池选择和实例级调度，以实现质量、时延、成本、稳定性和治理之间的平衡。

### 2.2 当前问题

当前系统在大模型调度场景中普遍存在以下问题：

| 问题类型 | 具体表现 | 业务影响 |
| --- | --- | --- |
| 请求与模型能力错配 | 简单请求进入高成本模型，复杂请求进入低能力模型 | 成本浪费、结果质量波动、用户体验不稳定 |
| 长短请求相互阻塞 | 长上下文、长输出请求占用大量算力与队列资源 | 短请求时延上升，长尾恶化 |
| 模型和实例调度逻辑割裂 | 模型选择、实例选择、风险控制分散在不同模块 | 决策路径复杂，难以统一维护和优化 |
| 动态运行态利用不足 | 路由层不理解语义，调度层不理解治理与能力边界 | 无法同时兼顾正确性和运行效率 |
| 调度不可解释 | 出现时延、质量、成本异常时缺少明确原因链路 | 难以复盘、难以迭代、难以审计 |
| 治理能力碎片化 | 风险请求、越权请求、异常请求处理口径不统一 | 合规风险和故障风险增加 |

### 2.3 传统方案的局限性

现有通用网关和普通负载均衡器难以直接满足大模型场景，主要原因包括：

- 传统网关擅长按接口、域名、请求头或显式规则转发，但无法稳定判断请求真实语义和能力诉求。
- 传统负载均衡擅长在实例间做流量分发，但通常不感知模型能力、上下文规模、会话连续性和缓存复用特征。
- 将语义选择、治理过滤和实例调度混在一起，会导致模块职责不清、决策不可解释、回放不可复现。
- 企业级 `LLM Gateway` 必须同时承担成本控制、服务质量、故障韧性和合规要求，仅依赖静态规则很难长期演进。

### 2.4 项目建设目标

本项目目标是建设一个面向 `LLM Gateway` 的语义路由与负载均衡子系统，使系统能够在毫秒级完成请求理解、候选模型池筛选、目标模型池选择和实例级调度，并提供完整的治理和反馈闭环。

具体目标如下：

- 建立 Gateway 内部统一的智能调度子系统，统一承接多模型、多实例、多供应商场景下的推理请求。
- 提升请求语义与模型能力的匹配度，减少资源错配和质量波动。
- 在池级和实例级同时优化时延、吞吐和 GPU 利用率。
- 在复杂企业场景下实现权限控制、风险治理、降级与审计闭环。
- 为后续多租户治理、策略优化、在线调优和异构后端纳管提供统一底座。

### 2.5 总体目标

从业务和平台视角看，本系统应同时实现以下四项目标：

1. 更高的模型能力匹配准确率。
2. 更低的平均时延和长尾时延。
3. 更高的算力与实例利用率。
4. 更强的可解释性、治理能力和可运营性。

## 3. 系统定位与建设边界

### 3.1 系统定位

本系统定位如下：

`LLM Gateway` 内部的智能语义路由与运行态调度子系统。

它不是完整的推理平台，不替代基础 API 网关，也不替代底层推理引擎或集群调度器。其核心职责是在请求进入推理执行链路之前，完成“该请求允许去哪里、应该优先去哪里、当前具体落到哪里”的分层决策。

### 3.2 本期建设范围

本期建设范围包括：

- 请求标准化、请求建模和语义信号提取；
- 统一 `Routing State` 构建；
- 基于能力、权限、风险和策略的候选池筛选；
- 模型池选择与备选路径生成；
- 目标模型池内的实例级负载均衡；
- 会话亲和、`Prefix / KV Cache` 亲和、长短请求隔离等大模型特化调度能力；
- 降级、熔断、摘除、故障切换和治理闭环；
- 可观测性、审计、解释、回放和反馈调优能力。

### 3.3 明确不在本期范围内的事项

以下事项不属于本项目当前建设范围：

- 模型训练、微调、蒸馏和离线评测体系建设；
- Prompt 工程编排、Agent 工作流编排或工具自动选择系统；
- 底层推理引擎的核心实现和 GPU 内核级调优；
- 独立的语义缓存产品化建设；
- 替代通用 API 网关已有的认证、鉴权、基础限流和南北向安全接入能力；
- 替代 `Kubernetes` 调度器或底层 GPU 资源编排器；
- 将 `Memory`、`RAG` 状态管理、`Responses` 状态持久化扩展为本阶段主建设目标。

### 3.4 设计原则

系统设计必须遵循以下原则：

- 语义正确性优先于运行态优化。请求首先必须被合法、合规地路由到正确能力域。
- 治理约束优先于成本、时延和吞吐优化。禁止为了性能收益突破权限和风险边界。
- 模型池选择与实例调度分层处理，避免职责混淆。
- 决策结果必须可解释、可审计、可回放。
- 优先建设可落地、可运营、可观测的能力，不以复杂的机器学习选择器作为一期前提。
- 优先保证主链路可用，再逐步提升性能上限和调优能力。

### 3.5 建设前提与约束

本期建设以前提条件成立为基础：

- Gateway 已具备基础接入、认证鉴权、协议适配和转发能力。
- 下游模型服务能够提供最低限度的健康状态、错误码和实例级统计信息。
- 平台侧能够维护模型池画像、策略配置和审计日志。
- 一期建设目标以控制面决策能力为主，不以改造底层推理引擎为前提。

## 4. 用户角色与典型场景

### 4.1 用户角色

| 用户角色 | 关注点 | 核心诉求 |
| --- | --- | --- |
| 业务系统 / Agent 平台 | 接入效率、稳定性、成本、输出效果 | 希望通过统一入口获得稳定、低成本、可预测的推理能力 |
| 平台产品 / 平台管理员 | 策略管理、模型池治理、容量管理 | 希望统一管理模型、策略、租户边界和演进路径 |
| 运维 / `SRE` | 可用性、容量、故障定位、指标观测 | 希望能够实时掌握请求流向、负载分布和异常原因 |
| 安全 / 合规团队 | 权限控制、数据边界、审计留痕 | 希望风险请求可识别、可阻断、可追踪 |

### 4.2 典型场景

系统至少需要支持以下典型场景：

1. 简单问答或轻量文本生成请求优先进入低成本、低时延模型池。
2. 长上下文、复杂推理、代码生成或高质量诉求请求进入高能力模型池。
3. 多轮会话在满足合法性和健康性前提下保持会话连续性，提升上下文复用效率。
4. 长请求与短请求分流，降低重任务对整体服务体验的拖累。
5. 当目标模型池过载、异常或不可用时，系统自动切换到备选路径或执行受控降级。
6. 不同租户、业务域和数据等级按独立策略进行路由和治理。
7. 对显式指定 `model` 或 `route_override` 的请求进行授权校验与兼容性校验。
8. 输出统一的解释信息、审计记录和观测指标，支持问题回放和持续调优。

### 4.3 现状痛点

从使用者视角看，当前系统的主要痛点包括：

- 调度逻辑偏静态，无法反映请求真实语义和运行约束。
- 模型池选择和实例调度缺少统一控制面，扩展和运维成本高。
- 决策结果不可解释，异常问题难以定位和归因。
- 大模型特有能力如会话亲和、缓存亲和和长短隔离未形成体系化支撑。
- 策略、治理、审计和反馈分散在多个模块中，难以形成闭环。

## 5. 系统设计总览

### 5.1 整体架构

系统整体采用“请求标准化 -> 信号提取 -> 状态构建 -> 决策匹配 -> 模型池选择 -> 实例级调度 -> 反馈回流”的分层控制面架构。

从处理链路看，北向请求首先进入 Gateway 接入层，由请求标准化与信号提取模块生成统一的请求事实和语义特征；随后由 `Routing State` 构建模块整合请求事实、身份、租户、治理和策略信息；在此基础上，决策匹配模块完成候选池过滤，模型池选择模块输出 `RouteDecision`，实例级负载均衡模块输出 `DispatchDecision`；最后，执行结果通过监控、审计和反馈机制回流至策略与阈值调整过程。

从职责划分看，策略与治理中心贯穿状态构建、候选过滤、池级选择和实例级调度四个关键阶段，但其职责仅限于提供规则、约束和执行策略，不直接替代具体业务模块的决策逻辑。

### 5.2 分层决策模型

系统从 Gateway 视角需要连续回答四个问题：

| 阶段 | 核心问题 | 决策对象 | 产出 | 不负责什么 |
| --- | --- | --- | --- | --- |
| 信号提取 | 这个请求是什么 | 请求本身 | 任务类型、复杂度、风险、会话等结构化信号 | 不直接选择模型池或实例 |
| 决策匹配 | 这个请求允许去哪里 | 路由规则和治理约束 | 合法候选模型池集合 | 不直接决定最终实例 |
| 模型池选择 | 这个请求应优先去哪里 | 候选模型池集合 | `RouteDecision` | 不决定具体实例 |
| 实例级调度 | 这个请求当前落到哪里 | 目标池内实例集合 | `DispatchDecision` | 不重做语义和治理判断 |

该分层模型的主要价值在于：

- 语义正确性与运行时最优性分层处理；
- 权限与治理边界不会被瞬时运行态覆盖；
- 池级选择与实例级调度边界清晰，便于问题定位、扩展和审计。

### 5.3 核心处理流程

系统主流程必须至少包括以下步骤：

1. 接收并校验北向请求，完成请求标准化。
2. 提取任务类型、复杂度、上下文规模、会话特征和风险信号。
3. 构建统一 `Routing State`。
4. 基于规则、能力边界和治理约束生成合法候选模型池集合。
5. 在候选集合内选择主选模型池和备选模型池，输出 `RouteDecision`。
6. 在目标模型池内部根据实例运行态输出 `DispatchDecision`。
7. 将请求转发到目标后端，并对结果进行日志、指标、审计和反馈采集。
8. 将运行结果回流至策略和阈值调优过程。

### 5.4 输入、状态、输出与反馈

| 类型 | 内容 |
| --- | --- |
| 输入 | 用户请求、Prompt / messages、控制参数、租户信息、策略配置、实例运行状态 |
| 状态 | `Routing State`、模型池画像、实例健康状态、会话状态、治理状态 |
| 输出 | `RouteDecision`、`DispatchDecision`、降级决策、审计记录、解释字段 |
| 反馈 | 时延、成功率、重试率、Fallback 结果、命中率、资源利用率、异常事件 |

### 5.5 模块划分

| 模块 | 核心职责 |
| --- | --- |
| 请求标准化与语义识别 | 对请求进行结构化理解，输出可计算的语义信号 |
| `Routing State` 构建 | 将请求事实、身份、策略和治理约束统一为单一决策上下文 |
| 决策匹配 | 基于规则和硬约束生成合法候选模型池集合 |
| 模型池选择 | 在合法候选域中选择主选池和备选池 |
| 实例级负载均衡 | 在已选池内选择具体执行实例 |
| 策略、降级与治理 | 管理优先级、授权、风险控制、Fallback、熔断和审计 |
| 可观测与反馈 | 记录指标、日志、回放链路和调优输入 |

## 6. 核心能力需求

### 6.1 请求标准化与语义信号提取

#### 目标

在请求进入决策前，对请求进行结构化理解，识别其任务类型、复杂度、上下文规模、会话特征和风险信号，为后续候选池筛选与模型池选择提供统一输入。

#### 必须满足的要求

- 系统 `MUST` 对北向请求做标准化处理，形成统一的请求对象。
- 系统 `MUST` 提取至少以下信号：任务类型、输入规模、上下文长度、复杂度等级、会话标识、风险标签。
- 系统 `SHOULD` 采用“轻量识别优先、重分析按需触发”的策略，避免不必要的高成本信号提取。
- 对于超长 Prompt，请求语义识别 `MAY` 使用仅用于信号提取的压缩视图，但 `MUST NOT` 改写实际发往下游模型的原始请求。
- 语义识别结果 `MUST` 结构化、可解释、可审计，不得依赖不可追踪的隐式逻辑。
- 信号提取模块 `MUST NOT` 直接输出最终模型池或实例决策。

#### 输入

- `prompt` / `messages`
- 显式控制字段，如 `model`、`routing_hint`、`session_id`
- 租户身份、权限信息和安全标签
- 会话上下文与历史摘要

#### 输出

- 任务类型标签
- 输入规模与复杂度标签
- 风险识别结果
- 会话连续性标签
- 请求工作负载画像

#### 验收关注点

- 相同输入在相同策略版本下的信号提取结果应稳定一致。
- 高风险请求必须输出明确风险标签，而不是静默进入普通路径。
- 长上下文请求的信号提取成本不能显著放大主链路时延。

### 6.2 `Routing State` 构建

#### 目标

将请求内容、控制参数、身份、租户约束、治理条件和策略偏好统一整理为单一的 `Routing State`，作为后续模型池选择和实例级调度的唯一决策上下文。

#### 必须满足的要求

- 系统 `MUST` 建立统一的 `Routing State`，避免不同模块重复解析请求并产生不一致结论。
- `Routing State` `MUST` 至少包含请求事实、控制参数、治理约束、会话状态和策略偏好五类信息。
- 系统 `MUST` 记录候选池裁剪原因，支持问题解释和审计。
- 硬约束过滤 `MUST` 在优化排序之前执行。
- `Routing State` 的生成过程 `SHOULD` 具备版本化能力，便于策略升级和回放复现。
- `Routing State` 的字段定义 `SHOULD` 具备稳定版本号，避免跨模块解释不一致。

#### 输出

- 统一 `Routing State`
- 可用候选池初始集合
- 不可用候选裁剪原因
- 解释与调试字段

#### 验收关注点

- 同一请求在同一策略版本下应生成一致的 `Routing State`。
- 裁剪原因必须能够指向具体的能力约束、权限约束或治理约束。

### 6.3 决策匹配与候选池过滤

#### 目标

根据规则、能力边界和治理约束确定“该请求允许在哪些模型池中被选择”，形成合法候选模型池集合。

#### 必须满足的要求

- 决策匹配 `MUST` 独立于模型池最终排序和实例调度。
- 系统 `MUST` 根据任务类型、复杂度、模态、上下文窗口、租户权限、合规标签和风险状态筛选候选池。
- 不满足权限、窗口、模态或治理要求的模型池 `MUST` 在该阶段被剔除。
- 显式 `model` 或 `route_override` 仅在授权且兼容时 `MAY` 生效。
- 若无合法候选模型池，系统 `MUST` 返回明确的无路由或策略拒绝结果，而不是隐式降级到任意模型。
- 决策匹配结果 `MUST` 能够输出规则命中顺序和优先级合并结果，以支持审计和回放。

#### 输出

- 命中的决策规则
- 合法候选模型池集合
- 被裁剪池及其原因

#### 验收关注点

- 不得出现未授权请求绕过治理约束访问高权限模型池。
- 同一请求命中多条规则时，必须有稳定的优先级合并机制。

### 6.4 模型池选择

#### 目标

在合法候选域中，根据质量、时延、成本和策略偏好选择主选模型池，并保留备选路径，为后续运行时切换提供依据。

#### 必须满足的要求

- 模型池选择 `MUST` 以合法候选池集合为输入，不得越过上游过滤结果。
- 系统 `MUST` 支持主选池和备选池输出，而不是仅输出单一路径。
- 系统 `MUST` 支持至少三类基础策略：时延优先、质量优先、成本优先。
- 系统 `SHOULD` 支持混合排序策略，例如质量/成本平衡、时延/质量平衡。
- 一期模型池选择 `SHOULD` 优先采用可解释规则和权重策略，不以黑盒学习型选择器作为基础前提。
- 池级选择结果一旦确定，实例层 `MUST NOT` 在未触发明确 Fallback 的前提下跨池调度。
- 当显式调用方偏好与平台级治理策略冲突时，系统 `MUST` 以治理策略优先。

#### 输出

- 主选模型池
- 备选模型池列表
- 选择原因与得分摘要
- `RouteDecision`

#### 验收关注点

- 复杂请求不应持续落入低能力池，简单请求不应长期滞留在高成本池。
- 备选池必须满足与主选池兼容的基本能力和治理约束。

### 6.5 实例级负载均衡

#### 目标

在目标模型池内部，结合实例健康、队列深度、资源利用率、会话亲和和缓存亲和信息，选择最合适的执行实例。

#### 必须满足的要求

- 实例级调度 `MUST` 只在已选模型池内进行，不得重做语义层判断。
- 系统 `MUST` 感知实例健康状态、队列深度、并发情况和基础资源利用率。
- 系统 `SHOULD` 支持会话亲和与 `Prefix / KV Cache` 亲和能力。
- 系统 `MUST` 支持长短请求隔离，避免重任务持续阻塞轻任务。
- 当实例异常、熔断或不可达时，系统 `MUST` 能进行摘除和重选。
- 亲和性与健康性冲突时，系统 `MUST` 以健康性和治理约束优先。
- 实例调度模块 `MUST` 记录触发重选、摘除或降级的直接原因。

#### 输出

- 目标实例
- 调度原因
- 是否命中亲和策略
- `DispatchDecision`

#### 验收关注点

- 热点实例不能长期过载而空闲实例长期无流量。
- 会话亲和和缓存亲和不能以牺牲系统稳定性为代价。

### 6.6 策略、降级与治理

#### 目标

对调度过程提供统一的策略模板、风险控制、权限校验、Fallback、熔断和审计能力，确保调度结果不仅可用，而且可控。

#### 必须满足的要求

- 治理约束 `MUST` 高于优化目标。
- 显式模型指定、`routing_hint`、`route_override` 的使用 `MUST` 接受权限和兼容性校验。
- 系统 `MUST` 支持模型池级和实例级两类 Fallback。
- 系统 `MUST` 支持实例摘除、熔断、恢复控制和异常事件留痕。
- 高风险请求 `MUST` 支持限制、强制降级、阻断或审计增强。
- 关键治理动作 `MUST` 记录触发规则、执行动作和最终结果。
- 治理动作 `MUST` 具备明确的生效范围、持续时间和恢复条件。

#### 输出

- 最终执行策略
- Fallback 路径
- 降级或阻断动作
- 审计记录

#### 验收关注点

- 未授权覆盖不能被静默放行。
- Fallback 不能突破租户、模态、窗口和合规边界。

### 6.7 可观测性、解释与反馈闭环

#### 目标

建立覆盖资源层、执行层、语义层和治理层的统一可观测体系，并将观测结果反向输入策略优化过程。

#### 必须满足的要求

- 系统 `MUST` 提供路由、调度、治理和资源四个维度的核心指标。
- 系统 `MUST` 为关键请求输出可解释的决策依据。
- 系统 `MUST` 支持对关键请求进行决策链路回放。
- 系统 `SHOULD` 将运行数据作为策略和阈值调优输入。
- 审计日志 `MUST` 能关联请求摘要、`Routing State` 摘要、规则命中结果、`RouteDecision` 和 `DispatchDecision`。
- 关键指标 `SHOULD` 支持按租户、模型池、策略版本和实例维度进行聚合分析。

#### 输出

- 指标看板
- 审计日志
- 路由解释字段
- 回放与复盘数据
- 策略优化输入

#### 验收关注点

- 关键路径请求必须可追溯、可解释、可复现。
- 观测数据不能只覆盖基础时延指标，还必须覆盖候选池选择和治理命中情况。

## 7. 功能需求清单

优先级定义如下：

- `P0`：一期必须具备，否则系统无法形成完整可用主链路。
- `P1`：应尽快补齐，直接影响调度效果和运营效率。
- `P2`：增强能力，用于提升长期优化能力和问题定位效率。

| 编号 | 功能项 | 需求说明 | 优先级 | 验收标准 |
| --- | --- | --- | --- | --- |
| `FR-01` | 请求标准化 | 将北向请求转为统一内部对象 | `P0` | 不同接口形态进入同一决策结构 |
| `FR-02` | 请求分类与工作负载画像 | 识别任务类型、复杂度、上下文规模和风险标签 | `P0` | 关键任务类型可稳定分类并输出结构化标签 |
| `FR-03` | 长上下文信号压缩 | 仅为信号提取生成压缩视图 | `P1` | 不改写真实下游请求且可降低识别成本 |
| `FR-04` | `Routing State` 构建 | 统一承载请求事实、身份、策略和治理状态 | `P0` | 决策链路不再依赖重复解析请求 |
| `FR-05` | 候选池过滤 | 按能力、权限、窗口、模态和治理约束筛选模型池 | `P0` | 非法池不得进入排序阶段 |
| `FR-06` | 决策匹配 | 根据规则和优先级生成合法候选域 | `P0` | 同一请求命中结果稳定且可解释 |
| `FR-07` | 模型池选择 | 输出主选池和备选池 | `P0` | 每个成功请求都能生成 `RouteDecision` |
| `FR-08` | 显式路由控制 | 支持 `model`、`routing_hint`、授权 `route_override` | `P1` | 非授权覆盖被拒绝，授权覆盖受兼容性校验 |
| `FR-09` | 实例级调度 | 在目标池内选择实例 | `P0` | 每个成功请求都能生成 `DispatchDecision` |
| `FR-10` | 会话亲和 | 支持 `session stickiness` | `P1` | 多轮会话在合法前提下具备稳定落点 |
| `FR-11` | `Prefix / KV Cache` 亲和 | 优先调度到高复用概率实例 | `P1` | 缓存亲和命中率可统计、可优化 |
| `FR-12` | 长短请求隔离 | 降低重任务拖累轻任务 | `P0` | 短请求长尾显著下降 |
| `FR-13` | Fallback 与故障隔离 | 支持摘除、熔断、切换和恢复 | `P0` | 异常实例不会持续接收新流量 |
| `FR-14` | 策略模板 | 支持质量优先、时延优先、成本优先和混合策略 | `P1` | 策略可配置、可切换、可审计 |
| `FR-15` | 治理与审计 | 支持权限、风险控制、审计留痕 | `P0` | 高风险请求处理有迹可循 |
| `FR-16` | 路由解释 | 输出关键决策原因与命中依据 | `P1` | 关键链路请求支持解释查询 |
| `FR-17` | 路由回放 | 支持关键请求的决策回放与复盘 | `P2` | 可以重建关键决策过程 |
| `FR-18` | 指标与监控 | 提供资源、时延、命中率、异常等全链路指标 | `P0` | 支持按池、按策略、按实例观测 |
| `FR-19` | 策略反馈调优 | 基于运行结果调整阈值和策略参数 | `P2` | 至少支持离线调优闭环 |
| `FR-20` | 配置管理 | 支持模型池画像、策略版本和阈值在线维护 | `P1` | 配置变更可追踪、可回滚 |

## 8. 接口与数据契约要求

### 8.1 北向接口范围

一期接口范围要求如下：

- 北向请求 `MUST` 以 `OpenAI API` 兼容接口为主。
- 一期优先支持 `HTTP/2 REST` 路径。
- 模型列表、健康检查、就绪检查和基础管理接口必须由 Gateway 暴露。
- 显式 `model` 字段如不在 Gateway 允许范围内，系统 `MUST` 在路由前拒绝，而不是交由下游兜底。

### 8.1.1 北向请求的最小字段要求

为保证决策链路可执行，一期请求至少应具备以下最小字段集合：

| 字段 | 说明 | 是否必须 |
| --- | --- | --- |
| `request_id` | 请求唯一标识；如上游未传入，由 Gateway 生成 | 是 |
| `model` | 显式模型或自动路由入口标识 | 是 |
| `messages` 或 `prompt` | 请求主体 | 是 |
| `tenant_id` | 租户标识 | 是 |
| `session_id` | 多轮会话标识 | 否 |
| `routing_hint` | 调用方路由提示 | 否 |
| `route_override` | 授权调用方的强制覆盖能力 | 否 |
| `preference` | 质量、时延、成本偏好 | 否 |

### 8.2 `RouteDecision` 契约

`RouteDecision` 至少应包含以下字段：

| 字段 | 说明 | 是否必须 |
| --- | --- | --- |
| `request_id` | 请求唯一标识 | 是 |
| `route_class` | 语义路由类别 | 是 |
| `target_pool` | 主选模型池 | 是 |
| `fallback_pools` | 备选模型池列表 | 是 |
| `matched_rules` | 命中的规则标识 | 是 |
| `reason_codes` | 关键决策原因码 | 是 |
| `governance_actions` | 触发的治理动作摘要 | 是 |
| `routing_confidence` | 路由置信摘要或信号质量摘要 | 否 |
| `debug_fields` | 调试或解释字段 | 否 |

要求如下：

- `RouteDecision` `MUST` 可被下游负载均衡模块直接消费。
- `RouteDecision` `MUST` 明确区分“合法候选域”和“最终主选池”。
- `RouteDecision` `SHOULD` 携带足够的解释信息，以支持审计和回放。

### 8.3 `DispatchDecision` 契约

`DispatchDecision` 至少应包含以下字段：

| 字段 | 说明 | 是否必须 |
| --- | --- | --- |
| `request_id` | 请求唯一标识 | 是 |
| `target_pool` | 所属模型池 | 是 |
| `endpoint_id` | 目标实例或端点标识 | 是 |
| `dispatch_reason` | 实例选择原因 | 是 |
| `affinity_flags` | 会话亲和或缓存亲和命中情况 | 否 |
| `load_snapshot` | 调度时的关键负载快照 | 否 |
| `fallback_level` | 是否发生重选或降级 | 否 |

要求如下：

- `DispatchDecision` `MUST` 能反映实例级决策原因。
- `DispatchDecision` `MUST NOT` 覆盖或篡改上游 `RouteDecision` 的合法性判断。

### 8.4 日志与审计字段要求

系统日志与审计数据至少应包含：

- `request_id`
- `tenant_id`
- 请求摘要或脱敏摘要
- 任务类型与复杂度标签
- 风险标签
- 命中规则列表
- 候选池列表与裁剪原因
- 主选池、备选池
- 实例选择结果
- 降级、熔断、阻断动作
- 路由耗时、调度耗时、总请求耗时
- 最终执行结果和错误码

所有日志与审计记录 `MUST` 满足最小必要原则，敏感字段必须支持脱敏或摘要化处理。

### 8.4.1 错误码与失败分类要求

为保证问题定位和自动化处置能力，系统 `SHOULD` 对失败场景进行标准化分类，至少包括：

- 请求格式错误；
- 权限或治理拒绝；
- 无合法候选模型池；
- 目标模型池不可用；
- 实例级调度失败；
- Fallback 失败；
- 下游执行失败；
- 观测或审计写入失败。

### 8.5 配置与策略管理要求

系统必须支持以下配置对象的管理：

- 模型池画像，包括能力、窗口、成本、模态、治理属性和适用租户；
- 策略模板与排序权重；
- 风险规则、权限边界和显式覆盖授权规则；
- 亲和性与隔离阈值；
- Fallback、熔断、恢复和摘除策略。

配置管理要求如下：

- 配置 `MUST` 支持版本化；
- 配置变更 `MUST` 可审计；
- 配置 `SHOULD` 支持灰度生效与快速回滚；
- 核心阈值变更 `MUST` 可追溯到责任人和生效时间。

## 9. 非功能需求

| 类别 | 要求 |
| --- | --- |
| 路由附加时延 | 在基线硬件和目标流量下，语义路由与调度带来的附加决策时延 `P99 SHOULD <= 10ms`，且不得显著放大主链路抖动 |
| 可用性 | 控制面不得存在单点瓶颈；月度可用性目标 `SHOULD >= 99.9%` |
| 扩展性 | 支持模型池、实例、策略和租户范围的动态扩展与调整 |
| 一致性 | 同一请求在同一策略版本下应产生稳定一致的路由结果 |
| 兼容性 | 一期兼容 `OpenAI API` 风格接口，并与 Gateway 现有接入层协同工作 |
| 可维护性 | 模型画像、规则、阈值和策略可在线维护并具备版本管理能力 |
| 可观测性 | 支持全链路日志、指标、事件、解释、回放与告警 |
| 安全性 | 支持权限隔离、敏感信息治理、越权阻断和审计留痕 |
| 可解释性 | 关键路由、降级和阻断决策必须可解释、可追溯 |
| 可恢复性 | 实例或模型池局部故障不能导致整条服务链不可用，系统应支持自动重选和受控恢复 |

## 10. 验收指标与成功标准

### 10.1 验收原则

一期验收不应只看“功能是否存在”，还必须关注“是否形成可运营的闭环”。因此，验收必须同时覆盖以下四个维度：

- 主链路功能完整性；
- 运行时性能与稳定性；
- 治理、审计和解释能力；
- 调优和迭代的可持续性。

### 10.2 指标目标与验收口径

以下指标为一期建议目标。若现网尚未形成统一基线，应在 `PoC` 或试运行阶段先完成基线测量，再锁定正式验收值。

| 指标 | 统计口径 | 当前状态 | 目标方向 |
| --- | --- | --- | --- |
| GPU 利用率 | 以目标模型池和核心实例集合的平均利用率统计 | 需在试运行阶段确认基线 | 较基线提升，建议提升不少于 `15% - 20%` |
| 平均时延 | 以核心北向请求的端到端平均时延统计 | 需在试运行阶段确认基线 | 较基线明显下降，建议下降不少于 `20% - 30%` |
| `P99` 时延 | 以核心北向请求的端到端 `P99` 统计 | 需在试运行阶段确认基线 | 显著收敛，建议控制在业务可接受范围内 |
| 模型能力匹配准确率 | 以抽样评审或离线标注结果统计 | 缺少统一统计 | 建立统一指标并持续提升 |
| `Prefix / KV Cache` 亲和命中率 | 以具备缓存复用条件的请求集合统计 | 缺少统一统计 | 建立统计体系并达到可观优化水平 |
| `Fallback` 成功率 | 以触发重选或降级的请求集合统计 | 缺少统一统计 | 核心异常场景下建议达到 `> 90%` |
| 路由解释覆盖率 | 以核心路径请求可回溯比例统计 | 缺少统一统计 | 核心路径请求 `100%` 可追溯 |
| 治理命中可审计率 | 以高风险、拒绝和阻断场景统计 | 缺少统一统计 | 高风险与阻断场景 `100%` 留痕 |

### 10.3 验收清单

系统交付时至少应满足以下验收条件：

1. 从请求进入到实例选定的主链路可稳定运行。
2. 至少支持一个完整的候选池过滤、池级选择、实例级调度和 Fallback 链路。
3. 关键请求具备解释、审计和回放能力。
4. 异常实例摘除、故障切换和恢复策略可验证。
5. 策略配置、模型池画像和关键阈值可在线维护并可回滚。
6. 指标体系能够区分语义层问题、池级选择问题和实例级调度问题。

### 10.4 验收方法建议

建议采用以下方式进行一期验收：

- 功能验收：依据 `FR` 编号逐项验证主链路、异常路径和治理路径。
- 性能验收：基于压测与回放流量验证时延、吞吐、资源利用率和长尾表现。
- 稳定性验收：通过实例下线、池级过载和治理命中场景验证 Fallback、熔断和恢复效果。
- 可运营性验收：验证解释、审计、回放、配置变更和指标看板的可用性。

## 11. 风险、依赖与待决问题

### 11.1 主要风险

| 风险 | 风险说明 | 应对方向 |
| --- | --- | --- |
| 模型能力边界不稳定 | 同一模型在不同任务上的表现可能波动 | 通过模型画像、灰度评估和策略回退控制风险 |
| 成本与质量冲突 | 高质量模型往往伴随更高成本和更高时延 | 采用池级策略模板和分层优化机制 |
| 长上下文预测误差 | 长上下文和长输出请求的资源估计难度高 | 建立更细粒度的工作负载画像和阈值调优机制 |
| 热点实例与缓存热点 | 亲和策略可能导致局部热点 | 通过限额、打散和健康优先原则平衡 |
| 数据不足影响调优 | 缺少高质量运行数据会限制策略优化效果 | 一期优先建设观测、解释和回放能力 |

### 11.2 关键依赖

系统落地依赖以下基础条件：

- 统一的指标采集体系，能够提供实例、GPU、时延和异常数据；
- 模型池画像体系，能够维护能力、成本、模态和治理属性；
- 策略中心或配置管理体系，支持版本化、灰度和回滚；
- 审计与日志平台，支撑问题复盘和治理留痕；
- 下游推理引擎和供应商接口能够提供稳定的健康与状态信息。

### 11.3 待决问题

以下问题需要在设计评审或一期前置方案中进一步明确：

1. 模型池的抽象粒度是按模型族、供应商能力档位还是部署拓扑划分。
2. 会话状态是否由 Gateway 独立维护，还是依赖外部状态服务。
3. 长短请求隔离采用逻辑隔离、物理隔离还是混合隔离策略。
4. `Prefix / KV Cache` 亲和所需的最小可观测数据集合由谁提供。
5. 一期是否需要为不同租户提供差异化的策略模板与容量上限。

## 12. 路线图与迭代计划

### 12.1 迭代原则

整体迭代遵循“先打通主链路，再提升效果上限，最后建设长期优化能力”的原则，避免在一期过早引入高复杂度机制。

### 12.2 阶段规划

| 阶段 | 时间建议 | 目标能力 | 退出标准 |
| --- | --- | --- | --- |
| `Phase 1` | `T0 - T+1 月` | 请求标准化、信号提取、`Routing State`、候选池过滤、基础实例调度、基础监控 | 主链路跑通，可稳定输出 `RouteDecision` 和 `DispatchDecision` |
| `Phase 2` | `T+2 - T+3 月` | 模型池选择、备选池、显式路由控制、基础 Fallback、治理审计 | 支持池级策略切换与实例异常切换 |
| `Phase 3` | `T+4 - T+5 月` | 会话亲和、缓存亲和、长短请求隔离、策略模板完善 | 性能和资源利用率明显改善 |
| `Phase 4` | `T+6 月及以后` | 路由回放、离线调优、在线策略优化、分阶段调度增强 | 形成稳定的反馈驱动优化闭环 |

## 13. 总结

语义路由与负载均衡系统不是传统 API 网关或普通负载均衡器的简单增强版本，而是 `LLM Gateway` 面向大模型场景的核心控制面能力。它要解决的不只是“请求分发到哪里”，而是“什么请求在什么约束下应该进入什么模型池，并在当前运行态下落到哪个实例执行”。

建设完成后，该系统将成为 Gateway 后续多模型接入、多租户治理、异构后端纳管、策略优化和运行效率提升的统一底座。它既决定系统的短期可用性，也决定平台的长期扩展能力和运营能力。
