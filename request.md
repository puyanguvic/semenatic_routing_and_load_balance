# LLM Gateway 语义路由与负载均衡子系统需求规格说明书（SRS）

## 1. 文档说明

### 1.1 编写目的

本文档用于定义 `LLM Gateway` 内部“语义路由与负载均衡子系统”的建设目标、职责边界、核心能力、功能需求、非功能需求、接口契约、验收标准和迭代范围，作为产品、架构、研发、测试、运维和安全团队的统一评审依据。

本文档关注“系统应当建设什么、达到什么标准、以什么边界落地”，而不是具体实现细节。设计、开发、测试和验收阶段均应以本文档为上位约束，保证需求、方案和交付结果的一致性。

### 1.2 适用范围

本需求文档适用于以下范围：

- 作为 `LLM Gateway` 内部控制面的子系统，负责对大模型请求进行语义理解、候选池筛选、模型池选择、实例级调度、治理约束执行和观测反馈。
- 面向北向调用方提供统一的推理入口，对南向异构模型服务、供应商接口和实例集群提供一致的调度抽象。
- 一期建设以 `OpenAI API` 兼容请求为主，服务 `chat/completions` 等主流推理调用路径。

### 1.3 目标读者

本文档面向以下角色：

- 产品经理：用于确认业务目标、建设边界和优先级。
- 架构师：用于评估职责拆分、接口边界和系统演进路径。
- 研发工程师：用于实现路由、调度、治理和可观测能力。
- 测试工程师：用于制定功能、性能和稳定性验证方案。
- 运维和 `SRE`：用于建设运行保障、容量规划和故障处置机制。
- 安全与合规团队：用于评审权限控制、风险治理和审计要求。

### 1.4 术语与约定

本文档中使用下列关键词表示要求强度：

- `MUST`：必须满足，否则视为不符合需求。
- `SHOULD`：推荐满足，除非有明确理由暂缓。
- `MAY`：可选能力，可根据迭代优先级决定是否实现。

核心术语定义如下：

| 术语 | 定义 |
| --- | --- |
| 语义路由 `Semantic Routing` | 基于请求内容、调用上下文和治理约束，决定请求“允许进入哪些模型池、应优先进入哪个模型池”的控制面能力 |
| 负载均衡 `Load Balancing` | 在已选模型池内部，根据实例健康、负载、亲和和运行态信息选择具体执行实例的能力 |
| 模型池 `Target Pool` | 具有相近能力边界、成本档位、治理属性或供应商特征的一组模型或实例集合，是语义路由的主要调度对象 |
| 路由状态 `Routing State` | 将请求事实、控制参数、租户身份、治理约束、会话上下文和策略偏好归一化后的统一决策上下文 |
| `RouteDecision` | 语义路由阶段输出的模型池级决策结果 |
| `DispatchDecision` | 负载均衡阶段输出的实例级调度结果 |
| 显式路由控制 | 调用方通过 `model`、`routing_hint`、`route_override` 等字段表达的显式意图 |
| 会话亲和 | 在满足合法性和健康性约束前提下，使同一会话优先落在稳定实例或稳定实例集合上的调度策略 |
| `Prefix / KV Cache` 亲和 | 优先将具有相似上下文前缀的请求调度到具备更高缓存复用概率的实例上 |

## 2. 项目背景与问题定义

### 2.1 背景

大模型服务已经从单模型、单集群、单场景调用，演进为多模型、多供应商、多租户、混合部署、长上下文、多模态和复杂治理并存的服务形态。在这一演进过程中，请求处理逻辑不再只是基础接入与转发问题，而逐渐转变为能力匹配、治理控制和运行态优化并存的调度问题。在此背景下，传统 API 网关以及传统四层、七层负载均衡方案已难以直接满足大模型场景的控制需求。

传统 Web 服务的请求成本通常较为稳定，调度关注点主要集中在连接数、请求数和实例健康状态；而在大模型推理场景下，请求的资源占用和执行代价受到多维因素共同影响，主要包括：

- 输入上下文长度和历史会话长度差异极大；
- 输出长度与推理复杂度波动明显；
- 请求是否涉及工具调用、多模态、结构化输出、代码生成或长链推理；
- 模型实例运行态受 GPU 利用率、显存占用、队列深度、`Prefill / Decode` 阶段切换和缓存命中状态影响较大；
- 企业级场景下还需同时满足租户隔离、权限控制、风险治理、审计追踪和成本控制要求。

因此，系统建设目标不应停留在流量均分层面，而应形成一套面向 `LLM Gateway` 的调度控制面，在请求执行前完成语义识别、合法候选筛选、模型池选择和实例级调度，并在质量、时延、成本、稳定性与治理之间建立可控平衡。

### 2.2 当前问题

当前大模型调度链路普遍存在以下问题：

| 问题类型 | 具体表现 | 业务影响 |
| --- | --- | --- |
| 能力匹配失衡 | 简单请求进入高成本模型，复杂请求进入低能力模型 | 成本浪费、结果质量波动、用户体验不稳定 |
| 工作负载相互干扰 | 长上下文、长输出请求占用大量算力与队列资源 | 短请求时延上升，长尾恶化 |
| 决策链路割裂 | 模型选择、实例选择、风险控制分散在不同模块 | 决策路径复杂，难以统一维护和优化 |
| 运行态信息利用不足 | 路由层不理解语义，调度层不理解治理与能力边界 | 难以同时兼顾正确性和运行效率 |
| 决策缺乏解释性 | 出现时延、质量、成本异常时缺少明确原因链路 | 难以复盘、难以迭代、难以审计 |
| 治理能力碎片化 | 风险请求、越权请求、异常请求处理口径不统一 | 合规风险和故障风险增加 |

### 2.3 传统方案的局限性

现有通用网关和通用负载均衡器难以直接支撑大模型调度控制，主要原因在于其能力模型与大模型请求特征并不匹配，具体体现在：

- 传统网关擅长基于接口、域名、请求头或显式规则做转发，但缺乏对请求语义和能力诉求的稳定识别能力。
- 传统负载均衡擅长在实例间分发流量，但通常不感知模型能力、上下文规模、会话连续性和缓存复用特征。
- 若将语义选择、治理过滤和实例调度混合在同一层处理，将导致职责边界不清、决策不可解释、回放不可复现。
- 企业级 `LLM Gateway` 同时承担成本控制、服务质量、故障韧性与合规要求，仅依赖静态规则难以支撑长期演进。

### 2.4 项目建设目标

本项目拟建设一个面向 `LLM Gateway` 的语义路由与负载均衡子系统，使系统能够在毫秒级完成请求理解、候选模型池筛选、目标模型池选择和实例级调度，并形成完整的治理与反馈闭环。

项目建设目标包括：

- 在 Gateway 内部建立统一的调度控制子系统，统一承接多模型、多实例、多供应商场景下的推理请求。
- 提升请求语义与模型能力的匹配准确度，降低资源错配与质量波动。
- 在模型池层与实例层同步优化时延、吞吐和 GPU 利用率。
- 在企业级场景下形成权限控制、风险治理、降级处置与审计追踪闭环。
- 为后续多租户治理、策略优化、在线调优和异构后端纳管提供统一基础。

### 2.5 总体目标

从平台建设和运行视角看，本系统应实现以下总体目标：

1. 提升模型能力匹配准确率。
2. 降低平均时延和长尾时延。
3. 提升算力与实例利用率。
4. 强化解释能力、治理能力和可运营性。

## 3. 系统定位与建设边界

### 3.1 系统定位

本系统定位为 `LLM Gateway` 内部的语义路由与运行态调度子系统，其职责是在请求进入推理执行链路之前，完成能力识别、边界裁剪、路径选择和实例落点选择。

本系统不是完整的推理平台，不替代基础 API 网关，也不替代底层推理引擎或集群调度器。其核心价值在于，围绕大模型请求建立分层决策控制面，将原本分散在接入、策略、调度和治理环节中的判断逻辑统一收敛。

### 3.2 本期建设范围

本期建设范围包括以下内容：

- 请求标准化、请求建模和语义信号提取；
- 统一 `Routing State` 构建；
- 基于能力、权限、风险和策略的候选池筛选；
- 模型池选择与备选路径生成；
- 目标模型池内的实例级负载均衡；
- 会话亲和、`Prefix / KV Cache` 亲和、长短请求隔离等大模型特化调度能力；
- 降级、熔断、摘除、故障切换和治理闭环；
- 可观测性、审计、解释、回放和反馈调优能力。

### 3.3 明确不在本期范围内的事项

以下事项不属于本项目当前建设范围：

- 模型训练、微调、蒸馏和离线评测体系建设；
- Prompt 工程编排、Agent 工作流编排或工具自动选择系统；
- 底层推理引擎的核心实现和 GPU 内核级调优；
- 独立的语义缓存产品化建设；
- 替代通用 API 网关已有的认证、鉴权、基础限流和南北向安全接入能力；
- 替代 `Kubernetes` 调度器或底层 GPU 资源编排器；
- 将 `Memory`、`RAG` 状态管理、`Responses` 状态持久化扩展为本阶段主建设目标。

### 3.4 设计原则

系统建设与设计应遵循以下原则：

- 语义正确性优先于运行态优化，请求应首先被合法、合规地路由至正确能力域。
- 治理约束优先于成本、时延和吞吐优化，任何性能收益都不得突破权限和风险边界。
- 模型池选择与实例调度分层处理，保持能力域判断与运行态判断的清晰边界。
- 决策结果应具备解释性、可审计性和可回放性。
- 一期优先建设可落地、可运营、可观测的基础能力，不以复杂学习型选择器作为前提。
- 优先保证主链路可用，再逐步提升性能上限和调优能力。

### 3.5 建设前提与约束

本期建设基于以下前提与约束：

- Gateway 已具备基础接入、认证鉴权、协议适配和转发能力。
- 下游模型服务能够提供最低限度的健康状态、错误码和实例级统计信息。
- 平台侧能够维护模型池画像、策略配置和审计日志。
- 一期建设目标以控制面决策能力为主，不以改造底层推理引擎为前提。

## 4. 用户角色与典型场景

### 4.1 用户角色

| 用户角色 | 关注点 | 核心诉求 |
| --- | --- | --- |
| 业务系统 / Agent 平台 | 接入效率、稳定性、成本、输出效果 | 通过统一入口获得稳定、低成本、可预测的推理能力 |
| 平台产品 / 平台管理员 | 策略管理、模型池治理、容量管理 | 统一管理模型、策略、租户边界和演进路径 |
| 运维 / `SRE` | 可用性、容量、故障定位、指标观测 | 实时掌握请求流向、负载分布和异常原因 |
| 安全 / 合规团队 | 权限控制、数据边界、审计留痕 | 对风险请求实现可识别、可阻断、可追踪的治理控制 |

### 4.2 典型场景

为支撑系统定位与建设目标，本系统至少需要覆盖以下典型场景：

1. 简单问答或轻量文本生成请求优先进入低成本、低时延模型池。
2. 长上下文、复杂推理、代码生成或高质量诉求请求进入高能力模型池。
3. 多轮会话在满足合法性和健康性前提下保持会话连续性，提升上下文复用效率。
4. 长请求与短请求分流，降低重任务对整体服务体验的拖累。
5. 当目标模型池过载、异常或不可用时，系统自动切换到备选路径或执行受控降级。
6. 不同租户、业务域和数据等级按独立策略进行路由和治理。
7. 对显式指定 `model` 或 `route_override` 的请求进行授权校验与兼容性校验。
8. 输出统一的解释信息、审计记录和观测指标，支持问题回放和持续调优。

### 4.3 现状痛点

从平台建设和运行视角看，当前系统主要存在以下痛点：

- 调度逻辑偏静态，无法反映请求真实语义和运行约束。
- 模型池选择和实例调度缺少统一控制面，扩展和运维成本高。
- 决策结果不可解释，异常问题难以定位和归因。
- 大模型特有能力如会话亲和、缓存亲和和长短隔离未形成体系化支撑。
- 策略、治理、审计和反馈分散在多个模块中，难以形成闭环。

## 5. 系统设计

### 5.1 设计目标与分层原则

本系统定位为 `LLM Gateway` 内部的调度控制面，其设计目标是在请求进入执行链路之前，完成语义理解、边界裁剪、路径选择和运行态调度的分层决策。对于大模型请求而言，模型能力匹配、租户与治理约束、成本与时延权衡、实例运行态优化并不属于同一类问题，如果在同一层中混合处理，系统将难以保持清晰的职责边界，也难以支撑解释、审计和后续演进。

基于这一判断，系统采用四层结构：请求理解与状态构建层、候选池过滤层、池级路由层、实例调度层。四层之间遵循由抽象到具体、由静态约束到动态优化、由能力域判断到执行落点判断的递进关系。前两层负责形成正确且合法的决策空间，后两层负责在该空间内完成策略选择与运行态落点选择。

系统设计遵循以下原则：

- 语义正确性优先于运行时优化，先确定请求需要什么能力，再讨论如何优化执行路径。
- 治理约束优先于性能目标，任何时延、成本或吞吐优化都不得突破权限、风险和数据边界。
- 池级路由与实例调度严格分层，避免能力域选择与实例运行态选择相互污染。
- 静态信息与动态信息分层处理，模型能力、租户边界、治理规则等稳定信息尽量前置，实例负载、健康状态、缓存命中等高频信息留在后层处理。
- 决策链路必须具备解释性，能够清晰说明请求为何被裁剪、为何选择某个模型池、为何落到某个实例。

### 5.2 四层设计逻辑

#### 5.2.1 请求理解与状态构建层

第一层负责将异构的北向请求转化为统一的内部决策语义。对系统而言，原始接口字段并不是最终决策对象，真正参与路由判断的是请求所表达的任务类型、复杂度、上下文规模、模态要求、会话连续性、风险特征以及调用偏好。因此，该层的首要职责是完成标准化和语义抽象，将接口差异消解为可供后续层稳定消费的统一事实集合。

在此基础上，该层进一步构建 `Routing State`，作为后续决策的统一上下文。`Routing State` 不只是请求内容的映射结果，还应整合租户身份、治理边界、策略偏好、会话状态和必要的环境信息。其设计意义在于，为整个决策链提供单一可信的输入源，避免不同模块重复解析请求并产生不一致结论。

从职责边界看，该层只负责形成“可决策请求”，不负责决定具体流向。它解决的是请求认知问题，而不是调度选择问题。将这两类问题分离之后，语义识别能力可以独立演进，策略调整也不会反向破坏请求理解逻辑。

#### 5.2.2 候选池过滤层

第二层负责基于 `Routing State` 定义合法的候选空间。其核心任务不是排序，而是边界裁剪。系统需要结合模型池画像、能力约束、租户边界、权限规则、风险规则和显式调用控制，对所有可达模型池进行一次严格过滤，仅保留在当前请求条件下具备合法性和兼容性的候选集合。

这一层的设计逻辑在于，将“是否允许进入某个能力域”的判断前置完成。窗口不足、模态不支持、权限不满足、数据等级不匹配、覆盖控制未授权等问题，本质上都属于硬约束，不应被后续的时延或成本偏好重新平衡。只有当候选域先被裁剪清楚，后续的路由优化才具备稳定边界。

因此，候选池过滤层是整个系统的边界收口层。它确保后续任何优化都只能发生在合法范围内，防止池级路由和实例调度承担本不属于它们的治理职责。对于无合法候选池的请求，系统应在此处直接给出明确结果，而不是将不确定性下推到后续链路。

#### 5.2.3 池级路由层

第三层负责在合法候选池集合内完成能力域级别的路径选择。经过前两层处理后，系统已经明确请求的语义特征以及合法边界，此时需要回答的问题不再是“能否服务”，而是“在可服务范围内，哪一类模型池最适合作为主路径”。这一层面向的是质量、时延、成本、稳定性等策略目标之间的权衡。

池级路由层的关键设计点在于，它只处理模型池这一抽象层级，而不直接介入具体实例的瞬时调度。模型池代表相对稳定的能力域和策略域，例如高质量池、低时延池、长上下文池或特定供应商池；实例则体现高频变化的运行态。将二者分离，可以保证池级决策围绕能力匹配和策略优先级展开，而不被局部瞬时负载扰动。

因此，该层应输出主路径和备选路径，并保证备选路径与主路径处于同样合法的治理边界之内。治理优先级必须高于调用方偏好，主备路径必须服务于同一决策逻辑。这样，后续的异常切换才不会演化为越界调度。

#### 5.2.4 实例调度层

第四层负责在目标模型池内部完成实例级执行落点选择。进入该层后，能力域已经确定，系统关注点从“选哪一类池”转向“在当前运行态下由哪个实例承接请求更合适”。这一层主要处理实例健康、并发压力、队列深度、资源利用率，以及会话亲和、缓存亲和、长短请求隔离等大模型场景下的运行时特征。

实例调度层的设计逻辑是，在不突破上层边界的前提下最大化执行效率与系统韧性。它应优先在既定模型池内部完成选择与重选，尽可能兼顾健康性、负载均衡和复用收益。当实例异常、过载或不可达时，应先在池内进行受控重选；只有在满足明确条件时，才可根据上层给定的备选路径执行进一步切换。

这一分层保证了两个层面的稳定性：一方面，上层已经确定的能力与治理判断不会被运行态策略改写；另一方面，运行态层可以独立围绕局部负载和故障情况做快速调整，而不影响池级路由的整体逻辑。

### 5.3 层间协同与控制闭环

四层设计形成的是一条逐层收敛的决策链，而不是若干并列模块的简单拼接。第一层完成请求语义抽象，第二层定义合法候选空间，第三层在合法空间内选择主备路径，第四层在既定路径内确定具体执行落点。每一层都以消化本层问题为目标，不将未完成的职责向下游转移。

这种协同方式本质上是对决策问题进行两次解耦。第一次解耦是将语义与治理边界同运行态优化分开，保证请求先被正确理解、正确裁剪，再讨论性能最优。第二次解耦是将能力域选择同实例选择分开，保证池级决策关注稳定策略，实例决策关注实时状态。通过这两次解耦，系统可以在保证语义正确与治理可控的前提下，逐步引入更复杂的优化能力，而不会破坏整体结构。

异常路径同样遵循分层闭环逻辑。若请求在候选池过滤层已无合法去向，系统应直接终止决策并返回明确结果；若池级路由已产生主备路径，实例调度层只能在既定路径内进行受控重选与切换；若执行阶段发生异常，观测、审计和反馈机制应能够逆向定位问题发生于语义识别、边界裁剪、池级选择还是实例调度。由此形成从请求进入、决策执行到异常复盘的完整控制闭环。

因此，本系统的设计重点不在于堆叠单点策略，而在于建立一套层次清晰、边界稳定、逻辑闭合的调度控制结构。其价值在于能够同时支撑语义路由正确性、治理可控性、运行效率优化以及后续策略演进。

## 6. 非功能需求

| 类别 | 要求 |
| --- | --- |
| 路由附加时延 | 在基线硬件和目标流量下，语义路由与调度带来的附加决策时延 `P99 SHOULD <= 10ms`，且不得显著放大主链路抖动 |
| 可用性 | 控制面不得存在单点瓶颈；月度可用性目标 `SHOULD >= 99.9%` |
| 扩展性 | 支持模型池、实例、策略和租户范围的动态扩展与调整 |
| 一致性 | 同一请求在同一策略版本下应产生稳定一致的路由结果 |
| 兼容性 | 一期兼容 `OpenAI API` 风格接口，并与 Gateway 现有接入层协同工作 |
| 可维护性 | 模型画像、规则、阈值和策略可在线维护并具备版本管理能力 |
| 可观测性 | 支持全链路日志、指标、事件、解释、回放与告警 |
| 安全性 | 支持权限隔离、敏感信息治理、越权阻断和审计留痕 |
| 可解释性 | 关键路由、降级和阻断决策必须可解释、可追溯 |
| 可恢复性 | 实例或模型池局部故障不能导致整条服务链不可用，系统应支持自动重选和受控恢复 |

## 7. 验收指标与成功标准

### 7.1 验收原则

一期验收不应只看“功能是否存在”，还必须关注“是否形成可运营的闭环”。因此，验收必须同时覆盖以下四个维度：

- 主链路功能完整性；
- 运行时性能与稳定性；
- 治理、审计和解释能力；
- 调优和迭代的可持续性。

### 7.2 指标目标与验收口径

以下指标为一期建议目标。若现网尚未形成统一基线，应在 `PoC` 或试运行阶段先完成基线测量，再锁定正式验收值。

| 指标 | 统计口径 | 当前状态 | 目标方向 |
| --- | --- | --- | --- |
| GPU 利用率 | 以目标模型池和核心实例集合的平均利用率统计 | 需在试运行阶段确认基线 | 较基线提升，建议提升不少于 `15% - 20%` |
| 平均时延 | 以核心北向请求的端到端平均时延统计 | 需在试运行阶段确认基线 | 较基线明显下降，建议下降不少于 `20% - 30%` |
| `P99` 时延 | 以核心北向请求的端到端 `P99` 统计 | 需在试运行阶段确认基线 | 显著收敛，建议控制在业务可接受范围内 |
| 模型能力匹配准确率 | 以抽样评审或离线标注结果统计 | 缺少统一统计 | 建立统一指标并持续提升 |
| `Prefix / KV Cache` 亲和命中率 | 以具备缓存复用条件的请求集合统计 | 缺少统一统计 | 建立统计体系并达到可观优化水平 |
| `Fallback` 成功率 | 以触发重选或降级的请求集合统计 | 缺少统一统计 | 核心异常场景下建议达到 `> 90%` |
| 路由解释覆盖率 | 以核心路径请求可回溯比例统计 | 缺少统一统计 | 核心路径请求 `100%` 可追溯 |
| 治理命中可审计率 | 以高风险、拒绝和阻断场景统计 | 缺少统一统计 | 高风险与阻断场景 `100%` 留痕 |

### 7.3 验收清单

系统交付时至少应满足以下验收条件：

1. 从请求进入到实例选定的主链路可稳定运行。
2. 至少支持一个完整的候选池过滤、池级选择、实例级调度和 Fallback 链路。
3. 关键请求具备解释、审计和回放能力。
4. 异常实例摘除、故障切换和恢复策略可验证。
5. 策略配置、模型池画像和关键阈值可在线维护并可回滚。
6. 指标体系能够区分语义层问题、池级选择问题和实例级调度问题。

### 7.4 验收方法建议

建议采用以下方式进行一期验收：

- 功能验收：依据第 5 章定义的四层职责与主链路逐项验证主路径、异常路径和治理路径。
- 性能验收：基于压测与回放流量验证时延、吞吐、资源利用率和长尾表现。
- 稳定性验收：通过实例下线、池级过载和治理命中场景验证 Fallback、熔断和恢复效果。
- 可运营性验收：验证解释、审计、回放、配置变更和指标看板的可用性。

## 8. 风险、依赖与待决问题

### 8.1 主要风险

| 风险 | 风险说明 | 应对方向 |
| --- | --- | --- |
| 模型能力边界不稳定 | 同一模型在不同任务上的表现可能波动 | 通过模型画像、灰度评估和策略回退控制风险 |
| 成本与质量冲突 | 高质量模型往往伴随更高成本和更高时延 | 采用池级策略模板和分层优化机制 |
| 长上下文预测误差 | 长上下文和长输出请求的资源估计难度高 | 建立更细粒度的工作负载画像和阈值调优机制 |
| 热点实例与缓存热点 | 亲和策略可能导致局部热点 | 通过限额、打散和健康优先原则平衡 |
| 数据不足影响调优 | 缺少高质量运行数据会限制策略优化效果 | 一期优先建设观测、解释和回放能力 |

### 8.2 关键依赖

系统落地依赖以下基础条件：

- 统一的指标采集体系，能够提供实例、GPU、时延和异常数据；
- 模型池画像体系，能够维护能力、成本、模态和治理属性；
- 策略中心或配置管理体系，支持版本化、灰度和回滚；
- 审计与日志平台，支撑问题复盘和治理留痕；
- 下游推理引擎和供应商接口能够提供稳定的健康与状态信息。

### 8.3 待决问题

以下问题需要在设计评审或一期前置方案中进一步明确：

1. 模型池的抽象粒度是按模型族、供应商能力档位还是部署拓扑划分。
2. 会话状态是否由 Gateway 独立维护，还是依赖外部状态服务。
3. 长短请求隔离采用逻辑隔离、物理隔离还是混合隔离策略。
4. `Prefix / KV Cache` 亲和所需的最小可观测数据集合由谁提供。
5. 一期是否需要为不同租户提供差异化的策略模板与容量上限。

## 9. 路线图与迭代计划

### 9.1 迭代原则

整体迭代遵循“先打通主链路，再提升效果上限，最后建设长期优化能力”的原则，避免在一期过早引入高复杂度机制。

### 9.2 阶段规划

| 阶段 | 时间建议 | 目标能力 | 退出标准 |
| --- | --- | --- | --- |
| `Phase 1` | `T0 - T+1 月` | 请求标准化、信号提取、`Routing State`、候选池过滤、基础实例调度、基础监控 | 主链路跑通，可稳定输出 `RouteDecision` 和 `DispatchDecision` |
| `Phase 2` | `T+2 - T+3 月` | 模型池选择、备选池、显式路由控制、基础 Fallback、治理审计 | 支持池级策略切换与实例异常切换 |
| `Phase 3` | `T+4 - T+5 月` | 会话亲和、缓存亲和、长短请求隔离、策略模板完善 | 性能和资源利用率明显改善 |
| `Phase 4` | `T+6 月及以后` | 路由回放、离线调优、在线策略优化、分阶段调度增强 | 形成稳定的反馈驱动优化闭环 |

## 10. 总结

语义路由与负载均衡系统不是传统 API 网关或普通负载均衡器的简单增强版本，而是 `LLM Gateway` 面向大模型场景的核心控制面能力。它要解决的不只是“请求分发到哪里”，而是“什么请求在什么约束下应该进入什么模型池，并在当前运行态下落到哪个实例执行”。

建设完成后，该系统将成为 Gateway 后续多模型接入、多租户治理、异构后端纳管、策略优化和运行效率提升的统一底座。它既决定系统的短期可用性，也决定平台的长期扩展能力和运营能力。
